# Go Core HTTP API Specification

Base URL (embedded mode): `http://localhost:{port}`
Base URL (server mode):   `http://{host}:{port}`

All request/response bodies are JSON unless noted.
Streaming responses use `text/event-stream` (SSE).

---

## Authentication

### Embedded mode
Every request must include the session key generated by Electron at startup:
```
X-Session-Key: {256-bit hex secret}
```
Missing or incorrect key → `401 Unauthorized`, process may exit.
Development mode (`CLANK_ENV=development`) skips this check.

### Server mode
```
Authorization: Bearer {jwt-token}
```
Missing or invalid token → `401 Unauthorized`.

### `GET /v1/health`

Returns server status. No authentication required.

**Response 200:**
```json
{
  "status": "ok",
  "version": "1.0.0",
  "mode": "embedded"
}
```

## Sessions

A session holds an isolated AgentLoop instance for a single chat.
One session per `chatId`. Sessions persist until explicitly deleted or the process exits.

---

### `POST /v1/sessions`

Create a new session.

**Request:**
```json
{
  "session_id": "chat-uuid",
  "agent": {
    "id": "agent-uuid",
    "name": "Assistant",
    "provider_id": "anthropic",
    "model_id": "claude-opus-4-6",
    "prompt": "You are a helpful assistant.",
    "required_skill_ids": ["skill-1"],
    "required_tool_ids": ["tool-1"],
    "required_mcp_server_ids": ["mcp-1"],
    "required_knowledge_base_ids": []
  },
  "config": {
    "providers": [
      {
        "type": "anthropic",
        "api_key": "sk-ant-...",
        "base_url": "https://api.anthropic.com",
        "is_active": true
      }
    ],
    "default_provider": "anthropic",
    "language": "en",
    "sandbox_config": {
      "default_mode": "sandbox",
      "sandbox_allow_list": [],
      "danger_block_list": []
    },
    "chat_permission_mode": "inherit",
    "chat_allow_list": [],
    "max_output_tokens": null
  },
  "skills": [
    { "id": "skill-1", "name": "Code Review", "system_prompt": "..." }
  ],
  "tools": [
    { "id": "tool-1", "name": "Search", "url": "...", "method": "GET" }
  ],
  "mcp_servers": [
    {
      "id": "mcp-1",
      "name": "filesystem",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": {}
    }
  ],
  "souls_dir": "/path/to/souls",
  "memory_dir": "/path/to/memory",
  "data_path": "/path/to/data"
}
```

**Response 201:**
```json
{ "session_id": "chat-uuid" }
```

**Response 409:** Session already exists for this `session_id`.

---

### `DELETE /v1/sessions/{session_id}`

Destroy a session and release all resources (stops running loops, kills MCP processes).

**Response 200:**
```json
{ "ok": true }
```

---

### `PUT /v1/sessions/{session_id}/config`

Hot-update session configuration without rebuilding the session.
Useful for updating skills, tools, or MCP servers mid-conversation.

**Request:** Same shape as `POST /v1/sessions` body (partial fields accepted).

**Response 200:**
```json
{ "ok": true }
```

## Running the Agent

### `POST /v1/sessions/{session_id}/run`

Execute the agent loop. Returns a streaming SSE response.

**Request:**
```json
{
  "messages": [
    { "role": "user", "content": "Explain this code" },
    { "role": "assistant", "content": "Sure, let me look at it..." },
    { "role": "user", "content": "What about the error handling?" }
  ],
  "attachments": [
    {
      "type": "file",
      "name": "main.go",
      "media_type": "text/plain",
      "data": "base64-encoded-content"
    }
  ],
  "agent_prompts": {
    "system_agent_id": "agent-uuid",
    "system_agent_prompt": "Additional context...",
    "user_agent_id": "user-agent-uuid",
    "user_agent_prompt": "User profile...",
    "assigned_task": null,
    "group_chat_context": {
      "agent_name": "Assistant",
      "other_participants": [
        { "id": "agent-2", "name": "Reviewer", "description": "..." }
      ]
    }
  },
  "injected_plan": null
}
```

**Response headers:**
```
Content-Type: text/event-stream
Cache-Control: no-cache
X-Accel-Buffering: no
```

**Response body:** SSE stream of chunk objects.
Each event is a JSON-encoded chunk as defined in `CHUNK_PROTOCOL.md`.

```
data: {"type":"context_update","metrics":{...}}

data: {"type":"agent_step","id":"step-init",...}

data: {"type":"text","text":"Hello"}

data: {"type":"text","text":", how can"}

data: {"type":"text","text":" I help?"}

data: {"type":"context_update","metrics":{...}}

data: {"type":"send_message_complete","stickyTargetIds":[]}

```

The stream ends after `send_message_complete` or `send_message_error`.

---

### `POST /v1/sessions/{session_id}/stop`

Stop the currently running agent loop for this session.

**Response 200:**
```json
{ "ok": true }
```

---

### `POST /v1/sessions/{session_id}/permission`

Respond to a `permission_request` chunk. Unblocks the suspended agent loop.

**Request:**
```json
{
  "block_id": "uuid-from-permission_request-chunk",
  "decision": "allow_chat",
  "pattern": "rm -rf /tmp/build"
}
```

| Field | Type | Values |
|---|---|---|
| `block_id` | string | Must match `blockId` from the `permission_request` chunk |
| `decision` | string | `"allow_chat"` — allow for this chat session only |
|            |        | `"allow_global"` — add to global allow list |
|            |        | `"reject"` — deny this specific tool call |
| `pattern`  | string | Command pattern to add to allow list (required for allow decisions) |

**Response 200:**
```json
{ "ok": true }
```

## Group Chat / Multi-Agent

Group chat runs are orchestrated by the caller (Electron `agent.js`).
Each agent gets its own session. The orchestrator:

1. Creates one session per agent in the group
2. Calls `POST /v1/sessions/{agent_session_id}/run` for each agent concurrently
3. Tags each chunk with `agentId` and `agentName` from the SSE stream
4. Scans accumulated text for `@mentions` to determine next-round agents
5. Repeats until no new mentions or `collaborationCancelled`

The Go server itself does NOT orchestrate multi-agent runs —
it executes one agent per session. Orchestration logic stays in the IPC bridge.

> **Note:** This may be revisited. Moving orchestration into Go is a future option
> but is out of scope for the initial migration.

## Context Compaction

### `POST /v1/sessions/{session_id}/compact`

Trigger standalone context compaction (when agent is not running).

**Request:**
```json
{
  "messages": [...],
  "enabled_agents": [...],
  "enabled_skills": [...]
}
```

**Response body:** SSE stream (same format as `/run`).
Emits `context_update` and `send_message_complete` when done.

## License (Server Mode Only)

### `GET /v1/license`

Returns current license status.

**Response 200:**
```json
{
  "valid": true,
  "customer": "acme-corp",
  "tier": "pro",
  "max_sessions": 10,
  "expires": "2027-01-01T00:00:00Z",
  "machine_id": "sha256:a3f8b2..."
}
```

**Response 402:** License invalid or expired.

## Machine ID (Server Mode Only)

### `GET /v1/machine-id`

Returns the stable machine fingerprint for license registration.
Also available as CLI: `clank-core --machine-id`

**Response 200:**
```json
{ "machine_id": "sha256:a3f8b2c4..." }
```

## Error Responses

All errors follow this shape:

```json
{
  "error": "human readable message",
  "code": "machine_readable_code"
}
```

| HTTP Status | Code | Meaning |
|---|---|---|
| 400 | `bad_request` | Invalid request body |
| 401 | `unauthorized` | Missing or invalid auth |
| 402 | `license_required` | License invalid/expired |
| 404 | `session_not_found` | Unknown `session_id` |
| 409 | `session_exists` | Duplicate `session_id` |
| 429 | `session_limit` | Max sessions for license tier reached |
| 500 | `internal_error` | Unexpected server error |

---

## SSE Stream Error

If an error occurs mid-stream, a `send_message_error` chunk is emitted
before the stream closes:

```
data: {"type":"send_message_error","error":"Provider 'anthropic' is missing apiKey"}

```

---

## Development Notes

- All endpoints accept and return `Content-Type: application/json`
- SSE streams use `\n\n` (double newline) to separate events
- Empty keepalive events (`: keepalive\n\n`) may be sent every 15 seconds
- Request bodies have a 10MB size limit
- SSE streams have no timeout (run as long as needed)
- All `session_id` values are treated as opaque strings (typically UUIDs)
