# Lessons Learned

Rolling log of rules derived from past mistakes. Appended to over time. Loaded into every session via `@LESSONS.md` from `CLAUDE.md`.

- One-time data migrations must be done by directly editing the data files on disk (e.g. `config.json`), not by adding migration logic to source code. Migration code in source is dead weight after first run, requires a restart to execute, and pollutes the codebase with logic that will never run again.

- `\b` (word boundary) in regex does NOT work for CJK (Chinese/Japanese/Korean) characters. CJK chars are non-`\w`, so `\b` between a CJK char and a space/punctuation (both non-`\w`) never fires. **Rule: always use `(?=\W|$)` instead of `\b` for name-boundary matching in `parseMentions`, `stripMentions`, and the `MessageRenderer` @mention highlighter.** Applies to any regex supporting non-Latin names.

- Group-chat collaboration (see also "Collaboration Loop — Iron Laws" in CLAUDE.md): (a) per-source `resolveAddressees` disambiguation so only addressed (not merely mentioned) agents reply each round; (b) `_applyChunk` in `chats.js` must NEVER touch any chunk tagged with `agentId` — all group chunk processing lives in `ChatsView.handleChunk` via `perChatStreamingMsgId`; (c) sharing a single bubble between agents causes duplicate placeholders, double writes, and `streaming=false` races that reroute text to the wrong bubble.

- NEVER run `npx vite build` (or any build command) after completing a task in this project. The user explicitly forbids it. Trust the code changes are correct; only run a build if the user explicitly asks.

- When using opacity/pointer-events for hover effects, always ensure buttons have explicit `z-index` AND the parent container has proper z-index stacking context. For sidebar action buttons use `z-index: 10-20`, for message action buttons use `z-index: 100`. Otherwise click events get intercepted by overlapping elements even when buttons appear visible.

- Non-obvious root cause behind Iron Laws #1/#2/#4: `ipcMain.handle` invoke-reply and `event.sender.send` agent chunks travel on DIFFERENT Electron IPC channels with NO ordering guarantee — the invoke can resolve BEFORE the final text chunks + `agent_end`, so `msg.content` is incomplete when `triggerAgentCollaboration` scans @mentions. That's why `waitForAgentEnd()` (poll `perChatStreamingMsgId` until cleared) is mandatory, why `collaborationCancelled` — not `!isRunning` — is the sole stop signal, and why `isRunning` must be re-asserted inside the collaboration loop.

- Roleplay truncation in `agent_end` must iterate segments — keep all non-text segments (tool_call, tool_result, image, permission), only truncate text segments to the trimmed length. Replacing `msg.segments` with `[{ type:'text', content:trimmed }]` wipes tool output and breaks the UI.

- Chinese → English migration: don't replace non-English text blindly. Classify first: (a) functional multilingual data to KEEP — i18n dictionaries, language-sensitive parsing, CJK stopwords/regex, hallucination blocklists; (b) non-functional UI copy — must route through `src/i18n/index.js` via `t('...')`, never hardcode translated literals or Unicode escapes in views/stores.

- After any structural data migration (e.g. flat config → `config.providers[]` array): (1) ask if backward compatibility is needed; (2) if yes, write a one-time migration of the data file, then delete ALL old code paths; (3) if no, delete old paths immediately. Never leave dual paths coexisting — they're invisible landmines for new users. Grep for old field names after migrating.

- Narrow-scope AgentLoop entry points — unified rule: any IPC handler that spawns an `AgentLoop` outside the chat path (doc edit, inline edit, utility model, task scheduler, IM bridge) MUST: (1) resolve provider/model via `_normalizeLoopConfig(loopConfig, agentId)` from `agentRuntimeUtils.js` — never reimplement provider resolution; (2) run `_validateLoopConfig(loopConfig)` so missing credentials surface cleanly instead of as a cryptic 404; (3) include a language directive sourced from `config.language` in the system prompt (LLMs don't reliably infer UI language from mixed-language input); (4) pass `excludedToolNames` to `ToolRegistry.loadForAgents` so narrow-scope features don't auto-inherit `update_memory` / `execute_shell` / `file_operation` etc. — principle of least privilege applies to LLM tools.

- AI Doc edit flow: (a) `_applyReplacement` must REFUSE when substring-find fails — never fall back to whole-file overwrite; distinguish `isWholeFileEdit = !targetText || targetText === currentFullContent`. (b) Auto-apply-latest-AI-edit hooks must scope to `messages.slice(beforeTurnCount)`, not the full message list — otherwise a `Revert` on a prior turn gets silently undone by the next send. (c) If a tool-based write to the CURRENT file happens (bypassing `<replacement>`), snapshot `activeFile.content` BEFORE the agent loop runs and attach a `toolEdit` metadata block so the UI can show a separate Revert pill — without the pre-edit snapshot there's no recovery path. The system prompt should steer the AI toward `<replacement>` for the current file and reserve `file_operation` for OTHER files.

- Chats store agent references by ID. ID resolvers for chat/message references must use `getAgentById(id) || defaultAgent`, not `id ? getAgentById(id) : defaultAgent`. The former handles both empty ID and dangling reference (agent deleted and recreated with new UUID); the latter silently breaks. Applies to `ChatGridPanel.vue`, `ChatsView.vue`, `useVoiceRecording.js`, and any future site resolving agent refs. Better: centralize via `resolveChatUserAgent(chat)` / `resolveChatSystemAgent(chat)` in the agents store.

- IPC subscription helpers in `preload.js` must use per-callback listeners: `const listener = (_, data) => cb(data); ipcRenderer.on(ch, listener); return () => ipcRenderer.removeListener(ch, listener)`. Never call `removeAllListeners` in a per-subscriber helper — that's a global hammer that silently starves sibling subscribers (e.g. `agent:edit-chunk` is shared by inline edit and AI Doc panel).

- Editors receiving programmatic writes (`editorContent.value = newText`) don't enter the native undo stack, so Ctrl+Z undoes the user's last keystroke instead of the AI edit. **Rule: maintain a separate AI-edit stack of `{ preContent, postContent, source, msgId }`. Intercept Ctrl+Z/Y/Shift+Z in capture phase but ONLY `preventDefault` when `currentContent === top.postContent`; otherwise pass through to native. Clear both undo/redo stacks on file switch.**

- Test hygiene: (1) When adding methods to a Pinia store consumed by components with tests, grep `src/**/__tests__/**/*.test.js` for `vi.mock.*<store-path>` and update the factory — stale mocks compile fine but crash on first call. (2) When a refactor removes a module, run `git log --diff-filter=D --summary | grep <Name>` and `grep -rn '<Name>' tests/ src/**/__tests__/` before committing — orphan test files that fail at suite-load report as "0 tests" and are easy to miss. Delete orphan tests and stale JSDoc in the SAME commit.

- Persistence policy: do NOT use Claude Code's auto-memory system at `~/.claude/projects/<slug>/memory/`. That path is per-device and opaque — writing there creates a split-brain with repo-tracked context. **Rule: ALL durable context (rules, lessons, project facts, collaboration preferences) goes into repo-tracked files (`CLAUDE.md`, `LESSONS.md`). If it's not worth putting in the repo, it's not worth persisting.**

- Vitest 4.x: (1) Test files must be ESM — `import { describe, it, expect, vi } from 'vitest'`; `require('vitest')` throws. (2) When the code under test is CJS and internally `require()`s a dependency, `vi.mock` often does NOT intercept that `require()`. Instead monkey-patch the CJS singleton: `const ds = require('../../lib/dataStore'); const orig = ds.paths; beforeAll(() => { ds.paths = () => fake }); afterAll(() => { ds.paths = orig })`. `module.exports = {...}` returns the same object to every caller, so method replacement is globally visible.

- `better-sqlite3` native binding is compiled for Electron's Node version (NODE_MODULE_VERSION=125 as of Electron 31), not local Node. Vitest runs in local Node, so directly `require('better-sqlite3')` from a test file fails with `ERR_DLOPEN_FAILED`. **Rule: SQLite-backed code must isolate its DB layer behind a thin facade and unit-test only the pure-logic helpers (parse/serialize/diff). DB integration is exercised at runtime, matching the existing `HistoryIndex` pattern. Do NOT `npm rebuild better-sqlite3` to make tests pass — it breaks Electron at runtime.**

- When migrating storage backends (e.g. .md files → SQLite), prefer **lazy auto-seed** over forcing the user to run a migration script. Implementation: pass the legacy directory to the new store, and on first read for a given (id, type) tuple, check for the legacy file and seed deterministically. Idempotent + one-shot via a `_seededAgents` Set. Keeps the migration script around for batch operations but makes the cutover seamless for end-users — no "where did my memories go?" if they forget to run the script. The legacy files remain on disk untouched as a safety net.

- Memory architecture (post-2026-04 cutover): single SQLite db at `{DATA_DIR}/memory/memory.db` with FTS5 trigram BM25 + vectra cosine semantic. There is no on-disk markdown blob format anymore — `MemoryStore` is the only source of truth. All read paths (`MemoryReadTool`, `systemPromptBuilder.readMemoryFile`, `ipc/voice.readMemoryFileSync`, `ipc/memory.readMemoryFileSync`, `ipc/agent.readMemoryFileSync`, `AnalyzeAgentTool`) delegate to `MemoryStore.readMarkdown(agentId, agentType)`. The legacy on-disk `souls/{type}/{id}.md` files were removed in the rename — sidecar JSON artifacts (`{id}.speech.json`, `.evidence.json`, `.harness.json`) live in `{DATA_DIR}/agent-artifacts/{type}/`. **Rule: NEVER add a new direct memory read path that bypasses the store — always go through the `MemoryStore` facade.**

- Group-chat roleplay truncation in `useChunkHandler.js` (`agent_end` handler) must require an explicit speaker-turn marker before cutting: pattern A `^\s*[:：]` immediately after the @-mention (agent impersonates with `@Name: ...`), or pattern B `\n\n` followed by `[Name]:` / `@Name` / `Name:` (paragraph break + new speaker). **Never hard-cap at N chars** — a bare `@Name` followed by the agent's own continuing prose is a salutation/turn-pass, not roleplay, and must be preserved verbatim. The previous 80-char unconditional fallback chopped legitimate sentences mid-word for any group message that started or ended with `@OtherAgent`.

- Built-in agent personas must each include their own LANGUAGE rule even though `systemPromptBuilder.js` injects a global LANGUAGE CONSISTENCY block. Reason: when the persona prompt itself is heavily English-coded (technical tokens, English imperatives, no language clause), low-temperature samplers like Qwen sometimes drift to English for planning/internal-monologue text — the persona's English bias outvotes the global rule. Clank's prompt has "Match the user's language. Use English for English users and Chinese for Chinese users."; DocMaster previously did not, and intermittently leaked English planning text in Chinese chats. **Rule: every built-in agent's i18n prompt must include a redundant per-persona language clause; relying on the global block alone is not enough.**

- **Update to the previous lesson (2026-04-29)**: per-persona language clauses are still NOT enough. Even with a Chinese language clause inside DocMaster's zh persona prompt, group-chat runs still produced fully-English replies when the user message + Clank's reply + @-mentions were all Chinese. The "match the user's most recent message" rule is too soft — heavily English-coded persona text (Markdown, YAML, TOC, footnotes, docstring) still wins under low-temp sampling. **New rule: `systemPromptBuilder.buildSystemPrompt` MUST prepend an `OUTPUT LANGUAGE — HARD RULE` block built from `config.language` ('zh' | 'en'), placed BEFORE the ABOUT THE USER block and persona identity. The directive must explicitly enumerate the failure modes ("overrides English-coded phrases in your persona", "overrides @mention identifiers", "overrides other agents' language in group chat") and carve out the only legitimate exceptions (explicit user request to translate; code keeps its programming language). Never rely on persona-internal language clauses alone — anchor to `config.language` at the system-prompt envelope level.**

- `config.json` credential encryption (post-2026-04): sensitive fields (`providers[].apiKey`, `smtp.pass`) are transparently encrypted at rest via `electron.safeStorage` (DPAPI / Keychain / libsecret), with `enc:v1:<base64>` prefix marking ciphertext. The encrypt/decrypt happens inside `dataStore.readJSON` / `writeJSON` / `readJSONAsync` / `writeJSONAtomic` keyed on `_isConfigFile(file)` — callers see plaintext, disk holds ciphertext. **Rule: when adding a new credential field to `config.json` (a new auth-bearing token, password, secret), update both `_decryptConfigSensitive` AND `_encryptConfigSensitive` in `electron/lib/dataStore.js`. Forgetting either side ships plaintext to disk silently. Also: never bypass dataStore by writing your own `readJSON`/`writeJSON` for `CONFIG_FILE` (e.g. local helpers in `main.js`, `im-bridge/message-router.js`) — delegate to `ds.readJSON` so encryption stays uniform.** `encryptString` is idempotent via the prefix check, so round-trip read/modify/write of unrelated fields stays safe.

- IM bridge self-chat dedup (Teams `48:notes` and any future self-mode adapter): tracking only `_sentMessageIds` populated AFTER `await _graphPost(...)` is racy — Microsoft creates the message at the START of the POST, but the ID isn't recorded until the response (~200–500ms later). If the polling timer fires inside that window, the bot's own outgoing reply gets ingested as a brand-new "user" message and rendered as a user bubble in the chat window. Symptom: in group chat, the second agent's reply (with `**AgentName**: ` prefix) re-appears as a user message stripped to `AgentName: ...` (markdown bold removed by `_stripHtml`), triggering another agent cascade. **Rule: in any IM adapter where the bot authenticates AS the user (Teams self-chat, Feishu p2p, etc.), pre-record the expected polled-back text BEFORE the POST and use it as a fallback dedup with consume-once semantics. ID-based dedup alone is not sufficient.** Critical detail for the content-hash fingerprint: Teams (and likely other Graph-style services) re-normalizes the HTML it stores — wrapping in `<p>`, adding whitespace between `<li>` siblings, etc. — so what comes back is structurally different from what we POSTed. The fingerprint MUST strip ALL whitespace (`/\s+/g → ''`), not just collapse to single spaces, otherwise lists/tables/multi-paragraph messages dedupe-miss and leak through.

- Streaming-placeholder cleanup in IM bridge `routeMessage` (`electron/im-bridge/message-router.js`): the function pre-writes an empty `{ role:'assistant', content:'', streaming:true }` row to disk + fires `im:agent-stream-start` BEFORE the AgentLoop runs. Every exit path must finalize OR remove that placeholder, otherwise it persists as an orphan and `chats.js::backfillChat` auto-resets `streaming:true → false` on next load, producing a permanent empty assistant bubble (assistant template renders the shell, but all v-if children are false because content/segments are empty). The original code only handled the success branch (`if (fullText) finalize else remove`) — the catch block did `continue` without cleanup, leaving one orphan per agent error. **Rule: any IM bridge code that creates a streaming placeholder must guarantee cleanup in EVERY exit path (success, error, abort), and pair it with a matching `im:agent-stream-end` so the renderer's in-memory bubble is also reconciled. Apply the same pattern to any future bridge that pre-writes placeholders.**

- IM bridge has TWO state planes: **disk** (`chats/{id}.json`) and **renderer in-memory** (`chatsStore.chats[].messages`). `finalizeMessage` / `appendMessage` / `removeMessage` only touch disk. The renderer in-memory bubble is mutated by THREE event handlers: `im:agent-stream-start` (push placeholder), `im:agent-chunk` (append text to `msg.content` + `msg.segments`), `im:agent-stream-end` (flip `streaming:false`, or splice if `removed:true`). **Trap**: in the success path, content lands in memory naturally because chunks fired during streaming. In the error catch path NO chunks fire, so `finalizeMessage` writes the error body to disk but the in-memory msg keeps `content: ''` → renderer shows an empty assistant bubble even though disk is correct (until the user reloads the chat). **Rule: any catch / abort path that wants to put content into the bubble (error indicator, partial result, anything) MUST emit a synthetic `im:agent-chunk` with `chunk: { type:'text', text: '...' }` BEFORE the `finalizeMessage` + `im:agent-stream-end` calls. Magic incantation order: `im:agent-chunk` (renderer appends) → `finalizeMessage` (disk) → `im:agent-stream-end` (renderer flips streaming). Same applies to any future event-driven IPC where there's a disk-side and a renderer-side state.**

- IM bridge polling must guard against `setInterval` re-entry. `setInterval(_poll, 5000)` does NOT await the async callback, so a slow `_poll` (Graph API jitter, multi-chat fetch, big chunk processing) can be re-entered while the previous tick is still running. Both ticks read the same stale `_lastCheckTime` → both ingest the same incoming msg → two `routeMessage` calls → duplicate user bubble + duplicate agent turn (the symptom is "I typed hi once but got two user bubbles + two full agent turns"). **Rule: every IM adapter polling loop needs (a) an in-flight boolean lock that makes overlapping ticks no-ops, AND (b) a TTL'd recently-ingested message-ID set as a defensive belt-and-suspenders guard. The lock prevents the common case; the ID set catches anything that slips through (Graph returning the same id twice, future refactors that bypass the lock, etc).** Symptom-without-fix to recognize: a "page flicker" in the chat window every send because `im:chat-updated` fires twice and the renderer's onChatUpdated sets `chat.messages = null` → reload → null → reload.

- Agents architecture (post-2026-04 cutover): `agents.json` and the `agent-artifacts/{type}/*.{speech,evidence,harness}.json` sidecars were merged into a single SQLite DB at `{DATA_DIR}/agents.db` with three tables (`agents`, `categories`, `import_artifacts`) plus `PRAGMA user_version = 1` and WAL. `electron/agent/AgentStore.js` is the only path — IPC handlers (`store:get-agents` / `store:save-agents`), `systemPromptBuilder.readSpeechDna`, `AnalyzeAgentTool`, IM bridge readers, `ipc/agent.js` lookups, `task-scheduler`, and `builtinSkills.seedBuiltinSkillsIntoAgents` all read through it. Builtin agents (`__default_system__`, `__doc_editor__`, builtin analyst) are seeded ONCE on first launch via the renderer's seed-on-missing logic in `src/stores/agents.js` — the old `mergeBuiltinSystemAgent` / `syncBuiltinDocEditorAgent` / per-config-change `watch()` re-merge mechanism is GONE. **Rule: NEVER reintroduce a re-merge or "refresh builtin from i18n" path on subsequent startups** — DB is SSOT, user customizations to builtin prompts/names/skills must persist across restart. Marketplace agent zip distribution (planned) will INSERT into the same tables, so the schema reserves `marketplace_id` / `marketplace_version` / `marketplace_signature` columns up front to avoid future ALTER TABLE. Bridge helper `ds.readAgentsCompat()` returns the legacy `{ agents: { items, categories }, personas: { items, categories } }` shape from AgentStore so legacy main-process callers (about a dozen of them) didn't need rewriting — keep using it for new code that wants that shape; for new code that just wants a flat agent list use `agentStore.getAll()` directly. **Trap that bit during this migration**: the renderer-facing IPC was only HALF the read paths — many main-process internals (`ipc/agent.js`, IM bridge, `agentRuntimeUtils`, `builtinSkills`, `task-scheduler`, `agent-recommendation/tool.js`) bypassed IPC and read `ds.readJSON(AGENTS_FILE)` directly. When migrating a JSON file to SQLite, grep for ALL readers (`grep -rn "AGENTS_FILE\|readJSON.*<file>"`) before deleting the file from disk — or you'll discover the bypass paths the hard way when the app silently shows zero agents.

- AgentLoop entry-point three-piece (Iron Law). Every code path that constructs a `new AgentLoop(...)` outside the canonical `agent:send-message` IPC handler — IM bridge `routeMessage` / `runWithBaseConfig`, future task scheduler, doc edit, inline edit, utility model — MUST satisfy ALL THREE pieces or interrupt/visibility silently breaks: **(1) Register in `activeLoops`** via `ipcAgent.registerLoop(\`${chatId}:${agentId}\` or \`${chatId}\`, loop, meta)` before `loop.run()`; `unregisterLoop(key)` in `finally`. `agent:stop` IPC iterates `activeLoops` by `chatId` prefix — non-registered loops are unstoppable. Also call `consumePendingStop(chatId)` between placeholder creation and `loop.run()` to honor stops that arrived during the race window. **(2) Reflect `chat.isRunning` in the renderer** by emitting a lifecycle event — IM bridge uses `im:run-started` / `im:run-ended` notified to the renderer, mapped to `chat.isRunning = true / false` in `ChatsView.vue`. Without this, the interrupt button (gated `v-if="isRunning"` in `ChatWindow.vue`) never appears and the user has no UI affordance to stop. **(3) Funnel all stop UX through `useInterrupt`** — never invent a parallel stop path. Single-source-of-truth contract: "OR `_hasActivity` across ALL streaming/waiting assistant msgs; if any active → mark + splice empties; if all empty → recall user msg + textarea." Single chat is the N=1 degenerate case of group chat. **Decision based on the LAST streaming msg alone is the recurring group-chat regression** — `pendingMsgs.some(_hasActivity)`, never `_hasActivity(lastAssistantMsg)`. Three further nuances learned the hard way: (a) For **sequential per-agent fan-out** (IM bridge iterates respondingIds), `loop.stop()` only kills the current loop — the for-loop will still construct a fresh AgentLoop for the next agent. Fix: a chat-level **cancellation flag** (`ipcAgent.markChatCancelled` / `isChatCancelled`, set by `agent:stop` IPC, cleared at routeMessage entry, checked at the top of every iteration to `break`). One Esc must terminate the whole turn, not race the for-loop. (b) When `removeMessage(chatId, msgId)` deletes the placeholder on disk, `im:agent-stream-end` ALSO needs a `removed: true` flag so the renderer splices it out of `chat.messages`. The default handler only flips `streaming: false`, leaving an empty assistant bubble in memory — that's the "ghost empty bubble" symptom, indistinguishable on screen from the disk-orphan bug. (c) `anyActivity` must scope to **the current turn** (since the last user msg), not just messages currently streaming. After a previous Esc has marked early agents (now `streaming: false` with `[Request interrupted by user.]`), a fresh empty placeholder for the next agent would otherwise flip the recall path and silently delete the user msg — even though earlier agents in the same turn already produced visible output.
